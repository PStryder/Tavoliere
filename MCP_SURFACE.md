# Tavoliere — MCP Research Surface (Future Extension)

## Concept

An MCP server exposing Tavoliere's research corpus and computed metrics as tools for AI-assisted research queries. This is NOT for gameplay (gameplay uses WebSocket). This is for analysis, exploration, and cross-session research.

## Why MCP

- Research queries are request-response shaped — natural MCP fit
- Lets researchers use Claude (or any MCP-capable AI) as a research assistant
- Natural language queries against structured corpus: "show me sessions where dispute density exceeded 10%"
- Backend endpoints already exist (research API); MCP server wraps them as tools

## Why NOT MCP for Gameplay

- Card games require real-time event streaming
- Consensus protocol depends on timely ACK/NACK (3-second objection window)
- MCP is request-response; WebSocket is the correct interface for play
- Polling via MCP would be fragile, slow, and token-wasteful

## Proposed Tool Surface

### Session Discovery
- `search_sessions` — filter by date range, deck recipe, participant composition (human/mixed/AI-only), model name/version, dispute density, duration
- `get_session_summary` — stats for a session: duration, action count, dispute count, participants, SPQ-AN profile
- `list_recent_sessions` — quick access to latest completed sessions

### Event Access
- `get_session_events` — full event log for a session, with optional filters (event type, seat, time range)
- `get_session_chat` — chat messages for a session, optionally filtered by channel (player/spectator)
- `find_disputes` — search for dispute patterns across corpus (by reason tag, resolution type, frequency)

### Metrics & Analysis
- `compute_spqan` — compute SPQ-AN profile (CE, RC, NS, CA, SSC) for a session or set of sessions
- `compare_models` — SPQ-AN metrics across model names/versions, with version pinning validation
- `get_friction_profile` — compression friction metrics for a participant or pair across sessions
- `get_norm_formation` — AUTO_ACK adoption curves, phase label entropy, convergence data

### Participant Research
- `get_player_history` — longitudinal data for a pseudonymized participant (requires longitudinal consent)
- `get_partnership_metrics` — coordination efficiency between specific partner pairs over time
- `compare_pair_types` — human-human vs human-AI vs AI-AI partnership metric comparison

### Export
- `export_dataset` — consent-filtered corpus export (NDJSON), scoped by session selection
- `export_metrics` — computed metrics as structured data (CSV/JSON)

## Implementation Notes

- MCP server wraps the same REST endpoints the Data Explorer UI uses
- Consent filtering is mandatory — MCP tools respect the same tier restrictions
- Identity data is never exposed through MCP; only pseudonymized research data
- The MCP server authenticates with a researcher-level API key

## Priority

Phase 5 — after research corpus is generating data worth querying. Requires:
- Phase 1-2 complete (playable + observable)
- Event log persistence (game history written to disk)
- SPQ-AN metric computation implemented
- Sufficient corpus size to make queries meaningful
